Part 2 — 설계 의도 · 자주 묻는 질문 — — —
왜 이렇게 만들었나
설계 의도와
자주 묻는 질문
Q1 에이전트를 왜 7개로 나눴나요?
컨텍스트 길이 한계 + 재실행 단위
CSV + EDA 보고서 + 분석 보고서 = 누적 시 50,000토큰+
—AI는 대화 전체를 메모리에 올리고 처리합니다. CSV 원본 + 분석 보고서 7개가 한 세션에 누적되면 메모리가 빠르게 차오릅니다.
—한도를 넘으면 앞 내용이 잘리고, 한도 안에서도 중간 내용을 놓치는 품질 저하가 생깁니다.
—에이전트로 나누면 각 단계가 자기에게 필요한 파일만 새 컨텍스트에서 읽습니다.
—한 단계만 바꾸고 싶으면 @dashboard만 재실행. 이전 단계 결과물은 그대로 재사용됩니다.
Q2 에이전트와 스킬은 뭐가 다른가요?
누가 하는가 vs 어떻게 하는가
—에이전트 — 역할·모델·권한이 고정된 AI 직원
—스킬 — 특정 작업의 실행 절차서
—/run-pipeline 실행 → 스킬 로드 → 오케스트레이터(Claude)에게 실행 순서 지시 → Claude가 에이전트 순서대로 실행
—나누는 이유 — '누가'와 '어떻게'를 분리해 각각 독립적으로 수정
Q3 에이전트를 분리하면 단점은 없나요?
컨텍스트 단절 · 토큰 비효율 · 단발성엔 과함
—컨텍스트 단절 — 파일에 명시된 것만 다음 단계로 전달, 뉘앙스가 사라짐
—토큰 비효율 — 중간 실패 시 그 전까지 쓴 토큰 낭비. .run_id 체크포인트로 완료 단계는 건너뜀
—단발성 분석엔 과함 — 단순 질문은 Claude에게 직접 하는 게 빠름
—반복 분석에 가치가 있습니다. Haiku×6 / Sonnet×1 구성으로 비용 최소화 가능.
