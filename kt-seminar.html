기존 BI 툴 (Tableau · PowerBI)
정해진 틀
안에서만
  • 차트 타입 — 라이브러리 지원 범위로 제한
  • 커스텀 시각화 — 개발자 필요, 오래 걸림
  • 데이터 전처리 — 별도 ETL 파이프라인
  • 인터랙션 — 필터·드릴다운 정도
  • 비용 — 라이선스 + 구현 기간
잘 만들어진 도구 — 그러나 제약이 있다
Claude Code + AI 파이프라인
제약 없이
만든다
  • Three.js · D3.js · Canvas · WebGL — 구현 제약 없음
  • 자연어로 설명하면 코드로 구현
  • 7단계 파이프라인이 CSV → 대시보드 자동화
  • 완전한 인터랙티브 — hover · filter · drill · animation
  • 비용 — API 사용료 + 시간 대폭 절감
"단순 막대그래프가 아니다"

기존 BI 도구 — Tableau · Power BI
이런 상황에서
압도적으로 강합니다
  • 전사 공유용 표준 리포트 — 비기술 사용자도 직관적으로 조회 가능한 안정적 리포팅 환경
  • 드래그앤드롭 인터페이스 — 코드 없이 차트 구성 가능, 현업 사용자 셀프서비스
  • 정해진 데이터 커넥터 (DB·DW·SaaS 40+종) — 대규모 운영 시스템 연동
  • 권한 관리 · 거버넌스 · 감사 로그 — 엔터프라이즈 보안 요건 충족

사전질문 답변 — Q3 · 대시보드 설계 필수 체크사항대시보드
설계 전 반드시 정의할 5가지
5W 프레임워크
이 5가지를 먼저 정하지 않으면,
아무리 예쁜 차트도 실무에서 쓸모가 없습니다.
W
Who — 누가 보는가?
임원 / 실무자 / 현장 담당자 — 독자가 달라지면 구조가 달라집니다
W
Why — 왜 봐야 하는가?
의사결정 / 이상탐지 / 정기 보고 — 목적이 KPI 선택을 결정합니다
W
What — 어떤 지표가 필요한가?
결과 지표 / 드라이버 / Actionable Input — 3계층 설계 (다음 패널 참고)
W
How — 어떻게 비교할 것인가?
전기 대비 / 목표 대비 / 업계 평균 — 맥락 없는 숫자는 의미가 없습니다
W
When — 언제 필요한가?
실시간 / 일간 / 주간 / 월간 — 갱신 주기가 아키텍처를 결정합니다
3-Tier 지표 체계
Level이 낮을수록 개선 방법이 명확해집니다
LEVEL 1
결과 지표 (후행)
매출 · CLV · 전환율 — 이미 일어난 일, 원인 추적 어려움
LEVEL 2
드라이버 지표 (선행)
배너 CTR · 앱설치 전환율 — 결과보다 먼저 움직여 예측·대응 가능
LEVEL 3
Actionable Input 지표 ← 이게 핵심
소재별 클릭전환율 · 캠페인별 노출 CTR — 낮으면 즉시 뭘 바꿀지 알 수 있음
대시보드 4단계 구조
이 4가지가 모두 있어야 의사결정이 가능합니다
① Quick Visibility
KPI 현황을 5초 안에
잘되고 있는지 안 되고 있는지를 첫 화면에서 즉시 파악. 색상 상태 표시 필수.
② Drill-down
원인 후보 탐색
카테고리별 · 세그먼트별로 내려가면서 "어디서 왜 이상한가"를 추적.
③ Action Items
다음에 뭘 해야 하는가
대시보드를 봤을 때 "그래서 뭐를 해야 해?"가 바로 나와야 합니다.
④ Raw Data
깊이 있는 탐색
원본 테이블 직접 조회 · CSV 다운로드 탭. 실무자가 직접 파고들 수 있게.

대시보드 · Q4
"질문이
차트를 결정한다"
차트 종류를 먼저 고르는 게 아니라, 답해야 할 질문이 먼저 나오면 차트 선택은 자연스럽게 따라옵니다. "이 차트가 답할 질문이 무엇인가"를 항상 먼저 써보세요.
꼭 지켜야 할 5가지
✓막대 차트 Y축은 반드시 0부터 시작
✓파이 차트는 도넛으로 대체할 것
✓강조 색상은 최대 1개 — 5개 다른 색은 아무것도 강조 안 함

WHY 01에이전트를 왜 7개로 나눴나요?
컨텍스트 길이 한계 + 재실행 단위
—CSV + 분석 보고서 7개가 한 세션에 누적되면 메모리가 빠르게 차오릅니다
—에이전트로 나누면 각 단계가 자기에게 필요한 파일만 새 컨텍스트에서 읽습니다
—한 단계만 바꾸고 싶으면 @dashboard만 재실행 — 이전 결과물은 그대로
WHY 02에이전트와 스킬은 뭐가 다른가요?
누가 하는가 vs 어떻게 하는가
—에이전트 — 역할·모델·권한이 고정된 AI 직원
—스킬 — 그 직원이 작업할 때 따르는 실행 절차서
—나누는 이유 — '누가'와 '어떻게'를 분리해 각각 독립적으로 수정 가능
WHY 03에이전트별로 모델을 다르게 쓰는 이유는?
—단순 전처리·요약 → claude-haiku (빠르고 저렴)
—HTML 대시보드 생성 → claude-sonnet (복잡한 코드 품질)
—Haiku×6 / Sonnet×1 구성으로 비용 최소화

Part 2 · Design System — CLAUDE.md vs Design Token— — —
방법 A — CLAUDE.md
자연어로 디자인 규칙을 설명
# CLAUDE.md 색상은 KT 빨간색을 메인으로 사용해줘. 폰트는 Pretendard, 본문은 읽기 좋게 넉넉한 줄간격으로 해줘. 카드는 흰색 배경에 그림자 약하게...
AI 토큰 비용 (세션당)
높음 — 파일이 길수록 선형 증가
주요 단점
×
매 세션 전체 로드 — CLAUDE.md는 모든 대화 시작마다 컨텍스트에 올라감. 파일이 길수록 비용 증가.
×
자연어 해석 차이 — "넉넉하게", "약하게"의 기준이 세션마다 달라짐. 일관성 보장 불가.
✓
빠른 시작 — 코드 없이 텍스트만으로 즉시 작성. 작은 프로젝트엔 여전히 유효.
방법 B — Design Token
수치를 코드 변수로 직접 정의
// theme.js — AI가 이 파일만 참조 primary: '#E8000D' background: '#F7F5F2' fontFamily: 'Pretendard' lineHeight: 1.8 borderRadius: 0
AI 토큰 비용 (세션당)
낮음 — 파일 크기 일정 유지
장점 / 단점
✓
토큰 소비 극소화 — 자연어 설명 없이 코드로 직접 참조. AI가 "읽을" 필요 없음.
✓
100% 일관성 — 같은 변수 참조 = 항상 같은 결과. 해석 차이 없음.
×
초기 세팅 필요 — 처음 한 번은 토큰 구조를 잡아야 함. 다음 슬라이드에서 프롬프트로 해결.
Part 2 · Design Token — 장단점— — —
장점 · Pros
바꾸기 쉬워지는 이유
일관성
한 곳만 바꾸면 전체 적용
Primary 색상 하나가 버튼·헤더·뱃지 모두에 연결. 브랜드 색상 변경 시 한 줄로 끝.
테마 전환
다크모드·브랜드 교체 즉시 반영
MUI createTheme()으로 테마만 교체하면 전체 UI가 한 번에 바뀜.
협업
디자이너 · 개발자 공통 언어
"Primary Blue"라는 이름을 공유. 디자인 파일과 코드가 같은 이름으로 동기화.
유지보수
하드코딩 찾아다닐 필요 없음
코드 전체를 검색하지 않아도 됨. 값이 어디에 쓰이는지 즉시 파악 가능.
단점 · Cons
감수해야 하는 비용
설계 비용
처음에 체계 잡는 시간 필요
어떤 단위까지 토큰화할지 결정해야 함. 너무 세밀하면 오히려 관리 부담 증가.
학습 곡선
팀 전체가 체계를 이해해야 함
토큰 이름만 보고 어떤 색인지 바로 알아야 함. 팀 온보딩 문서 필요.
추상화 위험
잘못 설계하면 오히려 복잡
토큰이 너무 많거나 계층이 깊어지면 어떤 값이 어디서 오는지 추적 어려움.
결론 — 3개 이상 페이지를 만드는 서비스라면
설계 비용을 감수할 가치가 있음
Part 2 · Design Token — 코드 비교— — —
Before / After
하드코딩 vs 토큰
❌ 하드코딩 — 수정 시 4곳 모두 변경
/* 브랜드 색상을 #0057B7로 바꾸려면? */ /* 버튼 */ .button { background: #C8001A; } /* 헤더 */ .header { border-bottom: 3px solid #C8001A; } /* 뱃지 */ .badge { color: #C8001A; border: 1px solid #C8001A; } /* → 4곳을 일일이 찾아서 바꿔야 함 파일 많을수록 누락 위험 ↑ */
✅ Design Token — 한 곳만 바꾸면 끝
/* 토큰 정의 — 딱 한 곳에서만 */ const theme = createTheme({ palette: { primary: { main: '#C8001A' } ↑ 여기 하나만 바꾸면 } }) /* 버튼 · 헤더 · 뱃지 모두 theme.palette.primary.main 참조 */ <Button color="primary">...</Button> sx={{ borderColor: 'primary.main' }} sx={{ color: 'primary.main' }} /* → 전체 자동 업데이트 누락 없음, 실수 없음 */
Part 2 · Design Token — 설계 방법— — —
처음 설계하는 사람을 위한 3단계
색상만 먼저, 나머지는 나중에
01
색상 4가지만 정의
Primary · Background · Text · Error
이 4가지로 대부분의 UI 커버 가능
primary: '#C8001A' background: '#FFFFFF' text: '#1C1A18' error: '#A1524A'
02
폰트 하나만 고르기
한국어: Pretendard 또는 Noto Sans KR
영어: Inter 또는 Geist
숫자·코드: JetBrains Mono
typography: { fontFamily: "'Pretendard', sans-serif" }
03
라이브러리 그냥 쓰기
MUI · shadcn/ui 같은 라이브러리는
토큰 체계가 이미 설계되어 있음
직접 처음부터 만들 필요 없음
→ 오늘 데모에서 쓴 방법
주의 — 완벽한 체계보다 일단 시작이 중요. 색상 4개로 시작해서 필요할 때 확장.
다음 단계 — 간격(spacing) · 테두리 반경(borderRadius) · 그림자(shadow) 순서로 추가.

결정한다

마케팅팀: 최근 7일 내 구매한 유저
→ 같은 질문, 다른 답 — AI는 어떤 정의를 써야 할지 모름
해결 방법
KPI 정의서 한 페이지로 시작
액티브 유저 = 최근 30일 내 1회 이상 로그인한 유저
제외: 테스트 계정, 내부 직원

Anthropic 내부 운영 원칙
AI는 초안 머신, 정의는 사람이

문서 초안 작성

사람이 해야 하는 것
